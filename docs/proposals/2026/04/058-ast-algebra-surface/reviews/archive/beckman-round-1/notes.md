# Beckman Scratch — equational reasoning

## Core objects and morphisms

Let:
- **V** = the bipolar vector space {-1, +1}^d (or ternary with Resonance, {-1,0,+1}^d).
- **T** = ThoughtAST — the free algebra generated by Atom literals under Bind/Bundle/Permute/Blend/Thermometer/etc.
- **E: T → V** = encode, a deterministic functor.

Key category claim (from FOUNDATION): E is supposed to be "composition-preserving." Let's check this rigorously.

## The MAP canonical skeleton

**Bind** (multiplicative group on {-1,+1}^d):
- Associative: (a*b)*c = a*(b*c) ✓ (pointwise)
- Commutative: a*b = b*a ✓
- Identity: the all-+1 vector 1_d ✓
- Self-inverse: a*a = 1_d ✓ (so inverse of a is a itself)
- Group law: (V, *, 1_d) is an abelian group of exponent 2 — an F_2 vector space of dim d

**Bundle** (threshold(sum)):
- Commutative ✓
- Has identity? Bundle([]) = 0-vector, but 0 is NOT in {-1, +1}^d — it's in the ternary {-1,0,+1}. Empty-bundle is degenerate in bipolar.
- **Associativity fails under threshold.**
  Consider x = +1, y = -1, z = +1 (d=1, scalar).
  - threshold(x+y) = threshold(0) — ambiguous (the proposals don't fix this).
  - Bundle([x, y, z]) = threshold(x+y+z) = threshold(1) = +1
  - Bundle([Bundle([x, y]), z]) = threshold(threshold(0) + 1) = depends on threshold(0)
  
  **If threshold(0) = +1:** Bundle([Bundle([+1, -1]), +1]) = threshold(+1 + +1) = threshold(2) = +1. OK matches.
  
  But for x=+1, y=-1, z=-1:
  - Bundle([+1, -1, -1]) = threshold(-1) = -1
  - Bundle([Bundle([+1, -1]), -1]) = threshold(threshold(0) + -1) = threshold(+1 + -1) = threshold(0) = +1 (with threshold(0)=+1 convention)
  - **MISMATCH.** -1 ≠ +1.

  **Conclusion: Bundle is NOT associative. It is n-ary, not binary-associative.**
  This is a well-known property of MAP-bundle — the sum preserves associativity, but the threshold does NOT distribute across intermediate thresholds.

  058-003 locks Bundle's signature as list-taking (good call), but **does not state the non-associativity law explicitly**. It should.

**Permute** (cyclic shift or fixed permutation P):
- P^k ∘ P^j = P^(k+j) — forms a cyclic group (Z/dZ for cyclic shift of length d, or arbitrary cycle for other P)
- Invertible: P^(-k) undoes P^k
- **Linearity**: Permute(Bundle(xs), k) = Bundle(map(Permute ·, ·, k), xs)?
  - 058-022 asserts: `Permute(Bundle(xs), k) = Bundle([Permute(x, k) for x in xs])` (linearity over Bundle)
  - This is TRUE at the vector level (permutation commutes with sum).
  - BUT the threshold? threshold(permute(sum)) = permute(threshold(sum)) iff permutation commutes with threshold. Since threshold is pointwise and permutation just reorders, yes — pointwise operations commute with permutations. ✓
- **Does Permute distribute over Bind?** Permute(Bind(a,b), k) = Bind(Permute(a,k), Permute(b,k))?
  - Pointwise: (P(a*b))[i] = (a*b)[P^{-k}(i)] = a[P^{-k}(i)] * b[P^{-k}(i)] = (Pa)[i] * (Pb)[i] ✓
  - So permutation distributes over pointwise bind too ✓
  - Proposals never state this as a law.

## Blend: is it really an algebra?

Blend(a, b, w1, w2) = threshold(w1·a + w2·b)

**Is Bundle a special case of Blend?** 
Bundle([a,b]) = threshold(a+b) = Blend(a, b, 1, 1). ✓

But variadic Bundle([a,b,c]) = threshold(a+b+c) = Blend(a, Blend(b, c, 1, 1), 1, 1)?
- Bundle([b,c]) = threshold(b+c)
- Blend(a, threshold(b+c), 1, 1) = threshold(a + threshold(b+c))
- NOT equal to threshold(a+b+c) (same non-associativity issue).

**So Bundle is NOT reducible to Blend via right-fold.** This means the "Blend subsumes Bundle at weights (1,1)" intuition in 058-002 is only valid for n=2. The proposal does NOT claim Bundle-elimination. Good.

**Commutativity of Blend:** Blend(a,b,w1,w2) vs Blend(b,a,w1,w2)?
= threshold(w1·a + w2·b) vs threshold(w1·b + w2·a)
= NOT EQUAL in general (unless w1=w2).
- 058-002 Q1 correctly notes this.

**Associativity:** Blend is binary, so no associativity question directly.

**Identity:** Blend(a, 0, 1, w) = threshold(a) = a (if a is bipolar). So 0-vector is right-identity. 
But 0-vector is NOT in the algebra's object space (T) without a null-atom. Can we construct 0? No — every Atom hashes to {-1,+1}^d. So identity is not directly expressible.

**Key law from 058-002 that needs checking:**
Amplify(x, y, 1) ≡ Bundle([x, y])?
- Amplify(x, y, 1) = Blend(x, y, 1, 1) = threshold(x+y)
- Bundle([x, y]) = threshold(x+y) ✓

Subtract(x, y) ≡ Difference(x, y)?
- Subtract(x, y) = Blend(x, y, 1, -1) = threshold(x-y)
- Difference(x, y) = Blend(x, y, 1, -1) = threshold(x-y) ✓ (IDENTICAL)

**Two names for identical operation — named for "reader intent."** This is a categorical smell. In a proper algebra, two expressions denoting the same morphism should have the same name. 058 knows this and pushes back in 058-004 and 058-019 but accepts the duplication.

**My verdict**: accept the duplication only if the cache canonicalizes them to share an entry. Otherwise, this is complection by aliasing.

## Orthogonalize — the computed-weight form

Orthogonalize(X, Y) = threshold(X - (X·Y)/(Y·Y) · Y)

**Claim (from 058-005)**: result is orthogonal to Y (dot product ≈ 0).

BUT THAT CLAIM IS ONLY TRUE BEFORE THRESHOLDING.

Before threshold:
X - (X·Y)/(Y·Y)·Y = X - proj_Y(X) = "X minus its Y-projection"
Dot with Y: (X - proj_Y(X))·Y = X·Y - (X·Y)/(Y·Y)·(Y·Y) = X·Y - X·Y = 0. ✓

AFTER threshold: the threshold(·) is a NON-LINEAR operation that can re-introduce a Y-component. Specifically:
  threshold maps each dimension to ±1. The Y-component of the thresholded result is NOT guaranteed to be 0.
  
Counterexample (d=4): let Y = [+1, +1, +1, +1] and X = [+1, +1, +1, +1].
  X·Y = 4, Y·Y = 4, coeff = 1.
  X - 1·Y = [0, 0, 0, 0].
  threshold([0, 0, 0, 0]) = [+1, +1, +1, +1] (with threshold(0)=+1 convention).
  Result·Y = 4, NOT 0.

**So the "orthogonal to y" claim in 058-005 is FALSE for the thresholded output.** It's true in real-valued space; false in bipolar. The proposal states this without qualification.

**This is a major categorical failure to flag.** The "invariant" claimed by Orthogonalize is not the invariant the operation actually produces.

Remedy: either (a) document that Orthogonalize's output is orthogonal-BEFORE-threshold and only APPROXIMATELY orthogonal after, or (b) drop the threshold and let Orthogonalize produce a non-bipolar (possibly ternary or real-valued) output. The latter changes the algebra's closure.

## Resonance — ternary output

Resonance(v, ref)[i] = v[i] if sign(v[i]) == sign(ref[i]) else 0

Produces {-1, 0, +1}^d — ternary output from two bipolar inputs.

**Does this break the algebra's closure?** The algebra's object space was {-1,+1}^d. Now we have {-1,0,+1}^d values. Does every downstream operation still work?

- Bind on ternary: if b[i]=0, then Bind(a,b)[i] = 0. Bind now produces ternary.
- **Bind's self-inverse property BREAKS on ternary**: Bind(Bind(a,b), b)[i] = a[i]·b[i]·b[i] = a[i] if b[i]≠0, 0 if b[i]=0. Not recoverable when b[i]=0.
  - 058-021 Q2 acknowledges this. Good.
- Bundle on ternary: threshold(sum of ternaries) — fine, output back to {-1, 0, +1} depending on threshold convention.
- Blend on ternary: same, threshold(w1·a + w2·b).
- Cosine on ternary: still well-defined (sum of products divided by norms).

**So ternary is mostly closed.** But Bind's reversibility law FAILS on ternary. This is a real algebraic fact that 058 does not state as a LAW:

  **Law: Bind is self-inverse in (V, *, 1_d) where V = {-1,+1}^d.**
  **Qualifier: the identity `Bind(Bind(a,b), b) = a` requires b[i] ≠ 0 ∀i.**

058-021 mentions this informally; it should be stated as a law.

## Sequential — composition with Bundle and Permute

Sequential([a,b,c]) = Bundle([Permute(a,0), Permute(b,1), Permute(c,2)])

**Associativity of Sequential:** 
Sequential([Sequential([a,b]), c]) = ?
  = Bundle([Permute(Sequential([a,b]), 0), Permute(c, 1)])
  = Bundle([Sequential([a,b]), Permute(c,1)])  (Permute by 0 is identity)
  = Bundle([Bundle([Permute(a,0), Permute(b,1)]), Permute(c,1)])
  = Bundle([Bundle([a, Permute(b,1)]), Permute(c,1)])

Sequential([a, b, c]) = Bundle([a, Permute(b,1), Permute(c,2)])

Note: c is permuted by 1 in the first, by 2 in the second. **These are NOT equal.** Sequential is NOT associative under itself.

This is fine — Sequential is not meant to be associative; it's a positional encoding. But this non-associativity should be flagged as a LAW ("Sequential does not associate — nesting produces different position encodings").

**Can we recover position via cleanup?**
Given Sequential([a,b,c]) and knowing we want position k, we Permute(·, -k) and cleanup. 058-026's nth does this. The noise from the other positions is bounded by Bundle capacity (~d/2·ln(K)).

## Then — composition with Chain

Chain(xs) = Bundle(pairwise-map Then xs)
Then(a,b) = Bundle([a, Permute(b, 1)])

Chain([a,b,c]) = Bundle([Then(a,b), Then(b,c)])
             = Bundle([Bundle([a, Permute(b,1)]), Bundle([b, Permute(c,1)])])

**Non-associativity again** — nested Bundles threshold separately.

Should Chain be:
  Chain'([a,b,c]) = Bundle([a, Permute(b,1), Permute(b,0), Permute(c,1)]) (sum all the terms, one threshold)
OR:
  Chain([a,b,c]) = Bundle([threshold(a + Permute(b,1)), threshold(b + Permute(c,1))]) (two thresholds, then one more)

The proposal's definition is the second — a Bundle of Thens. This is **NOT** equivalent to summing all constituent terms and thresholding once. The nested thresholds LOSE information.

Is this the intended semantics? Probably — each Then is itself a thought (a vector), and bundling them produces a superposition of pairwise-transition thoughts. But it should be EXPLICITLY stated that Chain is NOT equivalent to a single-threshold sum over all tokens.

## Duals

**Product / Coproduct:**
- Map (Bundle of Bind(k,v)) — this is product-like (a record)
- Coproduct = disjoint union / sum type = ENUM (058-030)
- Algebra has products (Map) but NO coproduct primitive at the vector level. Enum is a TYPE declaration, not a vector operation. Is there a coproduct vector operation missing?

**Fold / Unfold:**
- Bundle is a fold (reduces list to vector)
- Cleanup is partial unfold (vector to nearest of candidates)
- **Where is the general unfold?** Given a bundle and a list of roles, UNFOLD back to the list of fillers. This would be:
  ```
  unfold(bundle, roles) = [cleanup(Unbind(bundle, r), codebook) for r in roles]
  ```
- This COULD be a stdlib form but is never named. A natural missing dual.

**Bind / Unbind:**
- Bind is self-dual under bipolar (self-inverse). 058-024 argues Unbind as an alias. This is fine but the duality is trivial.
- Under ternary, Bind and Unbind DIVERGE. If ternary becomes a supported kind (Resonance), Unbind needs a real inverse, not just Bind again. 058-024 Q3 acknowledges this.

**Blend / unBlend:**
- Is there an inverse of Blend? Given Blend(a,b,w1,w2) = c, can we recover a (given b, w1, w2)?
- In general NO (the threshold destroys info). But approximate inverse exists:
  a ≈ (c - w2·b) / w1 (before threshold)
- This IS the structure of Orthogonalize! Orthogonalize(c, b) = c - (c·b/b·b) b — a specific "unBlend" when b's weight is projection-computed.
- So Orthogonalize is (a specific case of) "inverse Blend." Interesting — the duality isn't stated.

**Product / Sum as Bundle duality:**
- Bundle is sum/superposition. Is there a "meet" / conjunction form?
- Resonance IS a conjunction-like form: Resonance(v,ref) keeps dimensions where both agree.
- Complement: "Dissonance(v,ref) = v - Resonance(v,ref)" — 058-006 Q3 notes this as future stdlib.

## Type system

Built-ins: :Thought, :Atom, :Scalar, :Int, :String, :Bool, :Keyword, :Null, :List, :Vector, :Function, :Any

**Subtype hierarchy?** 058-030 Q8 recommends :Atom <: :Thought. OK.

What about:

- (:List :Atom) <: (:List :Thought)? This is **covariance question**. Proposal is silent.
  - Haskell/Scala perspective: :List should be covariant (read-only lists are covariant). 058 doesn't state variance.
- (:Function [:Thought :Thought] :Thought) vs (:Function [:Atom :Atom] :Thought)?
  - Function parameters are CONTRAVARIANT (can supply a function taking :Thought where :Atom is needed, because :Atom <: :Thought means :Thought function accepts more).
  - Wait, standard variance: f: A → B, if A' <: A and B <: B', then f: A' → B' as well (BY SUBTYPING rules).
  - Proposal is silent on function variance. This is a substantial missing property.

**Parametricity (free theorems):**
- For polymorphic `map: (:Function :T :U) × (:List :T) → (:List :U)`, parametricity gives free theorems (e.g., map f . map g = map (f . g)).
- 058-030 doesn't discuss parametricity or state any such laws.

## Cleanup's algebraic status

Cleanup(v, candidates) = argmax_{c in candidates} cosine(v, c)

**This is not algebraic in the monoidal/categorical sense.** It's a SELECTOR. It takes a list and returns one. The operation is:
- Non-deterministic if ties exist (proposal doesn't specify tiebreaking)
- Dependent on the similarity metric (cosine vs. Hamming vs. ...)
- Not a morphism in a clean category

**But it IS the algebra's BRIDGE to the outside world** — from noisy vector to clean candidate.

In category-theoretic terms: Cleanup is a CLOSURE OPERATOR against a finite subobject (the candidate set).

058-025 Q2 considers decomposing Cleanup into Similarity + Argmax. I agree — this would factor the categorical structure more cleanly. **Similarity** is a natural transformation V×V → Scalar; **Argmax** is a selector over a finite indexed family.

## Commute diagrams to check

Does the following diagram commute?

      encode
  T  ------→ V
  |          |
  map f      |  ?? what's the vector-level analog of map f ??
  ↓          ↓
  T  ------→ V
      encode

For a morphism f: T → T (a thought transformation), is there a fixed vector-level morphism g: V → V such that encode(f(t)) = g(encode(t))?

If f = Permute(·, k), then g is also Permute(·, k) on the vector. ✓
If f = (Bind r ·), then g is Bind(encode(r), ·). ✓ (linear map)
If f = (Bundle [·, x]), then g = Bundle([·, encode(x)]). ✓
If f = Amplify(·, y, s), then g = Blend(·, encode(y), 1, s). ✓
If f = Orthogonalize(·, y), then g = Orthogonalize(·, encode(y)). ✓

So most thought-level operations have vector-level analogs. encode is functor-like.

BUT:
- If f = (λt. eval(t)) — a reduction of an AST — then encode may not commute. eval changes the AST structure; encoding happens at the VALUE level.
- Specifically, encode(eval(t)) ≠ encode(t) in general.

This is an important property: **encode is NOT reduction-invariant.** An AST that hasn't been β-reduced has a different vector than its reduced form.

FOUNDATION's programs-are-thoughts claim relies on this NOT mattering (the vector IS the thought-as-AST), but for execution, the equivalence class of "thoughts that reduce to the same value" has many different vectors. This is a feature but deserves explicit statement.
